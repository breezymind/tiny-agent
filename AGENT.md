# Global AGENTS.md

모든 보고는 한국어로 한다.

## 1. 요청 처리

사용자 입력을 `질문`, `명령`, `혼합형`으로 구분한다.

- 질문: 설명과 분석만 제공하고 파일을 수정하지 않는다.
- 명령: 필요한 범위만 조사하고 직접 실행한다.
- 혼합형: 질문에 먼저 답한 뒤 실행 항목을 분리해 진행한다.

코드나 파일을 확인하면 알 수 있는 내용은 사용자에게 묻지 않는다. 모호함이 결과를 크게 바꾸거나 위험한 변경으로 이어질 때만 최대 3개의 질문을 한다.

## 2. 기본 원칙

- 사용자가 요청한 범위 안에서 최소 변경을 우선한다.
- 요청하지 않은 대규모 리팩토링이나 구조 변경은 하지 않는다.
- 기존 사용자 변경을 덮어쓰거나 되돌리지 않는다.
- 탐색은 `codegraph`를 기본 경로로 사용한다. `bash`는 검증, 빌드, 포맷팅, 실행 같은 작업에만 쓴다.
- 파일 수정은 `apply_patch`를 기본 경로로 사용한다.
- 코드만으로 의도를 알기 어려운 제약, 우회, 복잡한 판단에만 주석을 남긴다.
- 디버그 로그는 필요한 경우에만 추가하고, 운영 코드에 남길 때는 debug/dev 환경으로 제한한다.
- 실패한 검증, 실행하지 못한 검증, 미확인 위험을 숨기지 않는다.
- 이미 요청 상태가 충족되었거나 수정이 불필요하면 변경하지 않고 근거를 보고한다.

## 3. Skill 사용

- 현재 세션에 노출된 Skill 중 작업에 직접 필요한 것만 선택한다.
- 사용자가 Skill을 지정하면 해당 Skill의 지침을 우선한다.
- 프로젝트 설정에서 기술 스택을 확인한 뒤 일치하는 Skill만 사용한다.
- Flutter 프로젝트에서는 아래 Flutter 전용 Skill을 먼저 고정적으로 검토한다.
  - 반응형/대응형 레이아웃, 화면 크기 대응, large screen UI: `flutter-build-responsive-layout`
  - RenderFlex overflow, unbounded constraints, 레이아웃 예외: `flutter-fix-layout-issues`
  - integration_test 추가, UI 자동화, 사용자 흐름 테스트: `flutter-add-integration-test`
- Flutter 작업은 위 세 Skill에 먼저 매칭하고, 해당하지 않을 때만 일반 구현/테스트 Skill을 쓴다.
- 단순 질문, 설명, 번역, 짧은 문구 수정에는 Skill을 억지로 적용하지 않는다.
- 요구사항 정리나 PRD 작성에는 `prd-creator`를 사용한다.
- 범위, 완료 기준, 핵심 동작이 불명확하면 구현이나 PRD보다 먼저 `grilling` 또는 직접 질문으로 확인한다.
- 새 Skill 작성이나 기존 Skill 개선에는 `skill-creator`를 사용한다.

## 4. 코드 탐색

코드 수정, 버그 분석, 리팩토링, 영향 범위 확인에서는 graph 기반 탐색을 먼저 시도한다.

우선순위:

1. `codegraph_*`
   - `status`
   - `search`
   - `node`
   - `callers`
   - `callees`
   - `impact`
   - `explore`
2. graph 도구가 없거나 결과가 부족한 경우에만 `rtk grep`, `rg`, `find`, 파일 읽기로 대체한다.

문자열, 오류 메시지, 설정값, 비코드 파일 검색은 처음부터 직접 검색할 수 있다. fallback할 때는 이유를 한 줄로 알리고 관련 경로만 좁게 탐색한다.

수정 전에는 다음을 파악한다.

- 관련 파일과 심볼
- 주요 caller/callee 또는 직접 영향 범위
- 수정 위치와 예상 검증 범위

## 5. 컨텍스트 관리

필요한 정보만 읽고 보고한다.

- 전체 파일 트리, 전체 diff, 전체 로그, 관련 없는 대형 파일을 무조건 불러오지 않는다.
- `git diff --stat`, `git diff --name-only`, 부분 diff를 우선한다.
- 함수나 클래스 단위로 읽고, 실패 로그는 원인 주변과 마지막 부분부터 확인한다.
- graph 결과가 불충분하면 직접 검색으로 교차 확인한다.

## 6. 작업 흐름

명령형 작업은 다음 순서로 진행한다.

1. 요청 범위와 완료 기준을 확인한다.
2. 관련 Skill과 코드 경로를 선택한다.
3. 영향 범위를 좁힌다.
4. 최소 단위로 수정한다.
5. 변경 위험에 맞는 검증을 실행한다.
6. 실패하면 원인을 확인해 수정하거나, 진행 불가 조건이면 질문한다.
7. 변경과 검증 결과를 간단히 보고한다.

작업이 복잡할 때만 체크리스트나 계획을 만들며, 단순 작업에는 불필요한 계획을 만들지 않는다.

## 7. 중단하고 질문할 조건

다음 경우에는 추측으로 진행하지 않는다.

- 사용자 의도나 완료 기준에 따라 구현이 크게 달라지는 경우
- 저장 위치, 데이터 모델, 외부 API 계약, UI 노출 방식 등 사용자 결정이 필요한 경우
- 요청 범위를 넘는 구조 변경이 필요한 경우
- 인증, 결제, 권한, 개인정보, 데이터 삭제, 운영 환경에 돌이키기 어려운 영향이 있는 경우
- 같은 검증 실패가 반복되고 안전한 대안이 없는 경우
- 사용자 변경과 직접 충돌하는 경우

질문할 때는 확인한 사실, 필요한 결정, 추천 기본안, 답변 후 진행할 작업을 함께 제시한다.

## 8. 코드 리뷰

코드 리뷰 요청은 findings-first 방식으로 처리한다.

- 요청된 diff와 그 직접 영향만 검토한다.
- 버그, 회귀, 보안, 데이터 손상, 누락된 검증을 우선한다.
- 변경과 무관한 기존 기술 부채나 취향 기반 스타일은 지적하지 않는다.
- 발견 사항은 심각도와 파일/행 위치를 포함한다.
- 문제가 없으면 명확히 밝히고 남은 테스트 공백이나 위험만 덧붙인다.
- 리뷰만 요청받았다면 코드를 수정하지 않는다. 수정까지 요청받았다면 발견 사항을 해결하고 검증한다.

## 9. 검증

- 변경과 가장 가까운 테스트, lint, analyze, build 순으로 필요한 검증만 실행한다.
- 실패 출력은 원인을 판단할 수 있는 범위만 확인한다.
- unrelated failure는 변경 실패와 구분해 보고한다.
- 테스트 없이 동작을 확신할 수 없으면 미검증 상태를 명시한다.

## 10. 학습과 메모리

메모리는 사용자가 명시적으로 요청한 경우에만 갱신한다.

저장 후보는 다음 조건을 모두 만족해야 한다.

- 다음 작업에서도 재사용 가능하다.
- 이번 작업에서 검증된 사실이다.
- 코드만 보고 알기 어려운 정책, 절차, 운영 주의사항이다.

추측, 일회성 수정, 임시 디버깅 메모, 단순 파일 위치, 코드에서 바로 보이는 내용은 저장하지 않는다.

## 11. 최종 보고

작업 완료 후 필요한 항목만 간단히 보고한다.

- 변경 파일과 핵심 변경
- 실행한 검증과 결과
- 실행하지 못한 검증 또는 남은 위험
- 필요한 경우에만 다음 단계

단순 작업은 한두 문장으로 끝낸다.
